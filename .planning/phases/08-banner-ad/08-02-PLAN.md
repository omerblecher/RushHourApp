# Phase 8 Plan 02: Device Verification Checkpoint

## Goal

Verify on a real Android device (or running emulator) that the banner code from Plan 01 actually delivers the user-observable behavior described by BANNER-01 through BANNER-05. This is the only way to confirm BANNER-03 (dynamic padding) and BANNER-04 (cleanup on navigate away), since both depend on the native AdMob bridge firing real `SizeChanged` events and React unmount semantics on the device. It also re-confirms BANNER-01, BANNER-02, BANNER-05 in their visible form.

## Requirements Covered

- **BANNER-01** — Banner visible at bottom of GameScreen below ControlBar (visual)
- **BANNER-02** — Verify ADAPTIVE_BANNER + BOTTOM_CENTER via logcat / AdMob console
- **BANNER-03** — Bottom padding equals banner height; no content obscured (visual)
- **BANNER-04** — Banner gone after navigating away from GameScreen (visual)
- **BANNER-05** — Airplane-mode load failure leaves game fully playable, no error UI (visual)

<how-to-verify>
**BANNER-01 — Banner visible at bottom of GameScreen, below ControlBar**
1. From the launch screen, tap into a difficulty (e.g. Beginner).
2. Tap any puzzle to enter GameScreen.
3. Confirm a Google "Test Ad" banner is visible at the very bottom of the screen, sitting below the Reset/Hint/etc. ControlBar row.
4. Expected: banner is visible within ~2 seconds of GameScreen appearing.

**BANNER-02 — Adaptive banner at BOTTOM_CENTER**
1. While the banner is visible from step above, run in another terminal:
   `grep -i "banner\|admob" .planning/phases/08-banner-ad/logcat-08-02.txt | tail -50`
2. Look for log lines indicating banner load and position. The plugin typically logs `bannerAdSizeChanged` with width/height. Width should approximately equal screen width (adaptive). Height should be in the ~50-100px range.
3. If logcat is inconclusive, visually confirm the banner spans the full screen width and is anchored to the bottom edge.

**BANNER-03 — Bottom padding equals banner height; no content obscured**
1. Look at the ControlBar (the row with control buttons). It must NOT be hidden behind or partially covered by the banner.
2. There should be a visible gap (the dynamic padding) between the bottom of the ControlBar and the top of the banner — or the ControlBar should sit flush above the banner with no overlap.
3. Drag a vehicle — confirm the bottom row of the puzzle board is fully visible and tappable, not clipped by the banner.

**BANNER-04 — Navigating away removes the banner**
1. From GameScreen with the banner visible, tap the back button (or in-app navigation) to return to the puzzle list / difficulty screen.
2. Expected: NO banner visible on the puzzle list screen, the difficulty screen, the home screen, or the profile screen. The banner must only appear on GameScreen.
3. Re-enter a puzzle — the banner should reappear on GameScreen.
4. Navigate to ProfileScreen — confirm no banner.

**BANNER-05 — Failure does not affect gameplay**
1. Exit GameScreen.
2. Enable airplane mode on the device (Settings → Network → Airplane mode ON).
3. Re-enter a puzzle (GameScreen).
4. Expected:
   - NO banner appears (load fails silently due to no network)
   - NO error message, no toast, no red banner, no broken UI
   - The puzzle is fully playable: drag vehicles, complete the puzzle, see the win modal
   - The bottom of the GameScreen looks normal (zero padding, ControlBar at the very bottom)
5. Disable airplane mode and confirm a fresh GameScreen load brings the banner back (cleanup test).

**Logcat sanity check (optional but recommended):**
- After all five visual checks, stop the background logcat process: `kill %1` (or whatever the job number is).
- `grep -iE "exception|error.*banner|failedtoload" .planning/phases/08-banner-ad/logcat-08-02.txt`
- The only banner-related errors expected are `FailedToLoad` events during the airplane-mode test. There must be NO unhandled exceptions, no React error boundary trips, no crash logs.
